## Goal

Pick the dispatch shape that lets two or more concurrent `io.ready(from: fd,
interest:)` awaits on the same fd resolve correctly under the events strategy.
The choice also governs how the same guarantee is maintained in the completions
strategy (which already has it) so that the two actors stay architecturally
symmetric.

## P0 restated

`IO.Events.Actor.FDState` holds one `Async.Channel<Kernel.Event>.Unbounded.Ends`
per interest (`readEnds`, `writeEnds`, `priorityEnds`).
`IO.Events.Actor.fdMap` is keyed by `fd._rawValue`, so two concurrent
`io.ready(fd, .read)` calls on the same fd share one FDState. `awaitReadiness`
calls `fdState.readEnds.receiver.receive()`; the second caller to suspend on
that receiver trips the single-suspender precondition at
`Async.Channel.Unbounded.State.swift:186`:

```
precondition({
    if case .none = slot { return true }
    return false
}(), "Single-suspended-receiver invariant violated")
```

Reproduction: `Tests/IO Events Tests/IO.Events.Concurrent.Ready.Tests.swift`
(suite-disabled until the fix lands).

## Evidence: Completions already handles concurrent readiness correctly

A grep for `Async.Channel` in `Sources/IO Completions/` returns no matches. The
completions model dispatches entirely through:

- `IO.Completion.ID` — freshly generated by `IO.Completions.Actor.generateID()`
  for every operation.
- `IO.Completion.Operation.Storage` — a per-call class holding the completion
  slot and the dup'd descriptor.
- `CheckedContinuation<Void, Never>` — per-call, stored inside
  `IO.Completion.Entry`.
- `IO.Completion.Loop.entries: Dictionary<IO.Completion.ID, Entry>` — keyed by
  operation ID, not by fd.

Two concurrent `io.ready(fd, .read)` calls therefore produce two independent
SQEs, two independent CQEs, two independent continuations. Nothing is shared at
the per-fd level; nothing crashes.

The Completions twin test that originally documented this as a runtime
property was dropped — it hit the same `~Copyable`-into-`async let`
rejection as the Events test and would have needed an equally invasive
whitebox pivot (no single `awaitReadiness`-equivalent entry in the
completions actor). The invariant itself has migrated to
§"Completions Invariant" below; that section is the standing contract.

## Q1 reframed

The handoff frames the choice as "channel-per-awaiter vs fan-out queue". Reading
the crash path and the Completions reference shape surfaces a third option that
eliminates the Async.Channel layer in the Events actor entirely. The three
options are below; semantics-first arguments follow.

---

### Option A — Channel-per-awaiter

Each `io.ready` call allocates a fresh `Async.Channel<Kernel.Event>.Unbounded`
pair. The sender half is appended to a per-(fd, interest) list in the Loop's
`Registration`; the receiver half is held by the awaiter for the duration of
the call. `dispatchEvents` iterates the list and sends the event to every
sender; each receiver resumes, the channel is then dropped.

**Shape**:

```swift
extension IO.Event.Loop {
    struct Registration {
        var interest: Kernel.Event.Interest
        var readSenders: [Async.Channel<Kernel.Event>.Unbounded.Sender]
        var writeSenders: [Async.Channel<Kernel.Event>.Unbounded.Sender]
        var prioritySenders: [Async.Channel<Kernel.Event>.Unbounded.Sender]
    }
}

// In IO.Events.Actor.awaitReadiness(fdState, interest:)
var channel = Async.Channel<Kernel.Event>.Unbounded()
let sender = channel.sender
let ends = (consume channel).take().ends()
// ...register sender in registrations[id].readSenders...
arm(id: fdState.id, interest: interest)
_ = try await ends.receiver.receive()
```

**Pros**
- Mechanically smallest change: keeps the Loop's channel-based dispatch
  machinery intact.
- Cancellation path uses the channel's existing stop/close protocol.

**Cons**
- Heap-allocates a full Channel per call (state + pair of ends + internal
  buffer). The Completions path allocates less per call.
- The Loop now mutates the sender list on every ready call; requires
  bookkeeping to remove senders on awaiter cancellation to avoid send-to-closed
  noise.
- Leaves the Events actor architecturally out of step with Completions despite
  the two solving the same problem.

---

### Option B — Fan-out queue (reject)

Keep the one `Unbounded.Ends` per (fd, interest) in FDState. Extend the channel
state to allow multiple suspended receivers. Deliver one event to one receiver
(FIFO) and leave the remaining receivers queued.

**Cons (fatal)**
- Edge-triggered kernel semantics: a single ready-transition on the fd produces
  one event from the source. That event wakes one queued receiver. The other
  receivers wait for another edge that is not guaranteed to come — the drained
  reader might pull all bytes and leave the fd unreadable until the next write
  from the peer. The remaining awaiters deadlock.
- To avoid the deadlock, the dispatch would have to re-arm per receiver per
  event and effectively duplicate the event stream. That is broadcast delivery,
  not queue delivery; queue semantics are wrong for this domain.

Reject.

---

### Option C — Continuation-list on Registration (mirror Completions)

Remove the Async.Channel machinery from the Events path. Each `io.ready` call
stores a `CheckedContinuation<Void, IO.Event.Failure>` in a per-(fd, interest)
list held by the Loop's `Registration`. `dispatchEvents` resumes every
continuation in that list and clears it; the next `io.ready` call re-arms and
re-registers.

**Shape**:

```swift
extension IO.Event.Loop {
    struct Registration {
        var interest: Kernel.Event.Interest
        var readWaiters: [CheckedContinuation<Void, Never>]
        var writeWaiters: [CheckedContinuation<Void, Never>]
        var priorityWaiters: [CheckedContinuation<Void, Never>]
    }
}

// In IO.Events.Actor.awaitReadiness(fdState, interest:)
try await withTaskCancellationHandler {
    await withCheckedContinuation { (c: CheckedContinuation<Void, Never>) in
        register(c, id: fdState.id, interest: interest)  // append to list
        arm(id: fdState.id, interest: interest)
    }
    if cancellationObserved { throw IO.Event.Failure.cancellation }
} onCancel: {
    Task { [weak self] in await self?.removeWaiter(/* token */) }
}
```

(The concrete cancellation shape is a refinement, not a load-bearing choice.
See "Open sub-questions" below.)

**Pros**
- Architectural symmetry with `IO.Completions.Actor`: both actors dispatch via
  per-operation continuations keyed by a per-op identifier, and both deliver
  events directly to the continuation without an intermediate channel.
- Broadcast-to-all-awaiters on ready is the natural semantics. An fd's ready
  transition corresponds to exactly one dispatch pass that resumes every
  queued continuation and empties the list. Subsequent `io.ready` calls arm
  afresh.
- Removes ~70 LOC of channel plumbing: `FDState.readEnds` / `writeEnds` /
  `priorityEnds`, the sender halves in `Registration`, and the channel-close
  paths in `close()`/shutdown. The Events target loses its dependency on
  `Async` entirely except for whatever Wakeup channel remains.
- No per-call Channel allocation; a `CheckedContinuation` is the minimum
  scaffolding needed to suspend-and-resume.
- The `Kernel.Event` payload is not currently consumed by `awaitReadiness`
  (only `received != nil` is checked), so delivering `Void` is sufficient.

**Cons**
- Larger diff than Option A; touches FDState, Registration, dispatchEvents,
  close, and every actor call site that used to route through the channel
  ends.
- Cancellation must remove the exact continuation from the list. A token (the
  continuation's index at registration time is unstable under removal; use a
  fresh `IO.Events.Waiter.ID` or a class-backed record pointer) is needed.
  Mechanically similar to the flag-plus-cancel-coordinator pattern used in
  Completions.

**Recommended.**

---

## Why C rather than A

Three convergent arguments, in order of weight:

1. **Architectural symmetry.** Completions and Events solve the same concurrency
   problem (multiple suspended awaiters per fd) and should do so the same way.
   Completions has already established the continuation-per-operation dispatch
   shape and proven it correct; mirroring it in Events yields one mental model
   for both strategies. Option A keeps the two paths asymmetric for no
   additional capability.

2. **Semantics match the kernel.** Edge-triggered kqueue/epoll delivers one
   event per transition. The broadcast-on-dispatch pattern (wake every waiter,
   clear list, re-arm on the next call) matches that contract directly. A
   channel-based pattern forces the Loop to iterate a sender list and deliver
   the same event into N channels; this is equivalent in effect to Option C's
   broadcast but carries the Channel's state machinery for no benefit.

3. **Scope reduction.** Option C removes a dependency (Async from the Events
   target) and a class of state (the per-fd channel ends). Option A adds a
   list-of-senders field and a list-management path. The diff count favours C
   when you include the deletions it enables.

The cost — larger diff, explicit cancellation token plumbing — is a one-time
cost. Option A's smaller diff preserves the Async dependency and the
asymmetry indefinitely.

## Implementation sketch (if C is accepted)

Changes, scoped:

1. `Sources/IO Events/IO.Event.Loop.Registration.swift`
   - Replace `read` / `write` / `priority` sender fields with
     `readWaiters` / `writeWaiters` / `priorityWaiters: [Waiter]`, where
     `Waiter` holds the continuation and a cancellation identity.

2. `Sources/IO Events/IO.Events.Actor.swift`
   - Delete `FDState.readEnds` / `writeEnds` / `priorityEnds` (they disappear
     with the channel ends).
   - Replace `awaitReadiness` with a `withTaskCancellationHandler` +
     `withCheckedContinuation` body that appends a Waiter to
     `registrations[id].<interest>Waiters` and arms.
   - Add `registerWaiter`, `removeWaiter` helpers on the actor (both
     synchronous; actor isolation guards the list).
   - Update `close` to resume all waiters for the closed id with
     `.shutdownInProgress` instead of closing channel senders.

3. `Sources/IO Events/IO.Event.Loop.swift`
   - `dispatchEvents` becomes: for each event, look up the registration, drain
     all waiters for the matched interest, resume each with `.success`, clear
     the list. Re-arming is the responsibility of the next `io.ready` caller.

4. `Sources/IO Events/Exports.swift` / Package.swift
   - If the channel dependency goes dead, drop `import Async` from the Events
     target.

5. `Tests/IO Events Tests/IO.Events.Concurrent.Ready.Tests.swift`
   - Remove `.disabled` on the suite. The reproduction becomes the regression
     gate. Add a 3-awaiter variant if desired.

6. `Sources/IO Completions/IO.Completions.Actor.swift`
   - No functional change. Symmetry is achieved by the Events changes above;
     Completions already has the shape. Consider a brief doc-comment cross-
     reference to note that both actors follow the same dispatch pattern.

## Open sub-questions

These are refinements on top of the Option C choice; they do not gate the
Option A/B/C decision.

1. **Cancellation token shape.** Use a dedicated `IO.Events.Waiter.ID`
   (UInt64 counter, same pattern as `IO.Completion.ID`) and look up by ID, or
   use a class-backed `Waiter` record and compare by object identity? The ID
   approach is uniform with Completions; the class approach avoids a counter.
   Preference: ID, for uniformity.

2. **Continuation error channel.** `CheckedContinuation<Void, Never>` + a
   post-await cancellation check mirrors Completions. Using
   `CheckedContinuation<Void, Error>` with typed throws would let the Loop
   surface errors (shutdown, driver failure) directly, but typed throws on
   continuations is still limited in Swift 6.3. Preference: stay with
   `<Void, Never>` and check a flag, matching Completions.

3. **Sharing with Completions.** Is there a factorable `IO.Waiter` abstraction
   that both actors use, or is the duplication intentional because the two
   strategies resolve ID differently (kernel-issued CQE ID vs in-process
   counter)? Preference: keep them separate for now; factor only if a third
   strategy (Windows IOCP) wants the same shape.

4. **Broadcast order.** When multiple waiters are queued and an event fires,
   does the resume order matter? In practice the continuations resume on
   their respective task executors and any ordering is non-deterministic
   anyway. Preference: resume in registration order (list iteration) and
   document as non-load-bearing.

## Supervisor Decisions

Per [SUPER-015], the following decisions are now constraints on the remaining
work. Subsequent changes that deviate require explicit re-discussion.

### 3C-Q1: Option C adopted

**Decision**: Option C — continuation-list on `Registration`, mirroring the
completions strategy's per-operation dispatch shape. Options A and B are
rejected.

**Rationale cited by the supervisor** (ordered by weight):

1. **Broadcast-on-dispatch matches kernel semantics.** Edge-triggered
   readiness is "the fd transitioned — any of you may proceed." Waking one
   awaiter and leaving others queued would deadlock them waiting for a
   transition that already happened. Broadcast plus per-awaiter EAGAIN-retry
   is the standard reactor pattern.
2. **Architectural symmetry with Completions.** `Sources/IO Completions/`
   contains no `Async.Channel` usage; dispatch is per-operation via
   `IO.Completion.Entry.resolve`. Option C brings Events to the same shape.
   One mental model per strategy.
3. **Net LOC reduction.** Removing `Async.Channel` / `Unbounded.Ends`
   machinery from the Events target is a structural simplification, not a
   stylistic one.

**Explicitly rejected reasoning for Option A**: Smaller diff is not the
optimization target. Per-call channel allocation on the hot readiness path
costs per-op, and cementing the architectural asymmetry with Completions
means every future `_ready` tweak has to be implemented twice with different
semantics.

### 3C-Q2: Waiter identity and continuation shape

**Decision (1)**: `IO.Events.Waiter.ID = Tagged<IO.Events.Waiter, UInt64>`.
The ecosystem Tagged convention for IDs has landed — `IO.Completion.ID` is
already `Tagged<IO.Completion, UInt64>`
(`Sources/IO Completions/IO.Completion.ID.swift:28`). Events adopts the same
shape. The allocating counter is actor-isolated, monotonically increasing,
mirroring `IO.Completions.Actor._nextID`. No reuse concerns at `UInt64`
range.

**Decision (2)**: `CheckedContinuation<Void, Never>` with a post-await flag
check. Matches `IO.Completions.Actor.awaitOperation`'s "resume with void,
caller re-checks state" shape. Avoids the Copyable constraint that a
payload-carrying continuation would impose and preserves the cancel-handshake
hook point for the `onCancel` handler.

### Citation in the implementation

The commit message and any PR description must cite the governing decision
by name so the review trail is clear.

### 3C-Q1 REVISED: Per-call channel (supersedes Option C)

**Decision**: Option C is reversed. The adopted shape is **per-call channel**
(refined Option A): each `io.ready` call creates its own transient
`Async.Channel<Kernel.Event>.Unbounded`. The sender is appended to a
per-interest list in `IO.Event.Loop.Registration`; the receiver is local to
the awaiter. `dispatchEvents` iterates the sender list and broadcasts.
`close` iterates and closes each sender.

**Governing rule**: [IMPL-060] — "When the ecosystem provides a type,
operation, or infrastructure at any layer, it MUST be used via dependency
rather than reimplemented." Option C replaced ~70 LOC of `Async.Channel`
dispatch with ~150 LOC of hand-rolled continuation management. That is
reimplementation of producer-consumer dispatch — precisely what
`Async.Channel` provides.

**Additional rationale**:
- Events dispatch is channel-shaped (producer → consumer). Completions
  dispatch is lookup-shaped (per-op ID → keyed entry → resolve). Forcing
  Events into Completions' shape violates [IMPL-INTENT] — hand-rolled
  continuations express mechanism; channels express intent.
- Per-call channel allocation (one `Async.Channel.Unbounded` per `io.ready`)
  is negligible against kernel work (one `kevent`/`epoll_wait` per wakeup
  dwarfs a Swift heap allocation).
- No new types required. No `IO.Events.Waiter.ID`, no `Tagged` convention
  adoption, no continuation-list helpers. `Async.Channel` already provides
  cancellation semantics (close sender → receiver returns nil).

**3C-Q2 is superseded**: the `Tagged<IO.Events.Waiter, UInt64>` ID type and
`CheckedContinuation<Void, Never>` pattern are no longer needed under the
per-call channel shape.

## Completions Invariant

Option C's architectural argument rests on the claim that
`IO.Completions.Actor` is already structurally correct for concurrent
readiness awaits on the same fd. That claim was originally backed by a twin
reproduction test in `Tests/IO Completions Tests/`, but that test hit the
same Swift 6.3 `~Copyable`-into-`async let` rejection and would have needed
an equally invasive whitebox pivot (no single `awaitReadiness`-equivalent
entry point exists in the completions path; each `ready` call already
allocates its own id/storage/entry/continuation). The twin was dropped; its
value has migrated here.

Deleting a test does not delete the contract it documented. This section is
the contract.

### The invariant

**Stated**: `IO.Completions.Actor` dispatches readiness awaits per-operation,
not per-fd. Two or more concurrent `io.ready(from: fd, interest:)` calls on
the same fd produce independent in-flight records — no shared per-fd
channel state, no single-receiver precondition to violate.

### Current evidence (2026-04-15)

Verbatim grep against the commit containing this note:

```
$ rg -l 'Async\.Channel' 'Sources/IO Completions/'
(no matches)
```

Specifically:

- `Sources/IO Completions/IO.Completions.Actor.swift:80` allocates a fresh
  `IO.Completion.ID` via `generateID()` for each `read` / `write` / `ready`
  call. The ID is monotonic, actor-isolated, never reused.
- `Sources/IO Completions/IO.Completions.Actor.swift:174` — `ready(from:
  interest:)` constructs a per-call `IO.Completion.Operation.poll(...)` whose
  `Storage` holds that ID and a freshly duped descriptor.
- `Sources/IO Completions/IO.Completion.Entry.swift:31` — `Entry` carries its
  own `CheckedContinuation<Void, Never>`; `resolve(with:)` resumes that
  exact continuation once and then consumes the entry.
- `Sources/IO Completions/IO.Completion.Loop.swift:86` —
  `entries: Dictionary<IO.Completion.ID, Entry>` is keyed by operation ID,
  never by fd. Two concurrent ready calls on the same fd occupy two
  independent entries under different IDs.

There is no `Async.Channel` in the completions dispatch path — neither as a
per-fd receiver nor anywhere else. The wakeup path (`IO.Completion.Wakeup.Channel`)
is a different construct: a single-writer/single-reader poll-thread
interrupt, not an event-delivery channel.

### What would violate the invariant

Any future change that re-introduces shared per-fd dispatch state in
`Sources/IO Completions/` is a violation. Specifically:

1. Adding an `Async.Channel` (or any shared channel primitive) to
   `IO.Completions.Actor`, `IO.Completion.Loop`, or `IO.Completion.Entry`.
2. Keying `IO.Completion.Loop.entries` by fd (raw or typed) rather than by
   `IO.Completion.ID`, or introducing a secondary fd→entry index that
   collapses concurrent awaiters.
3. Sharing `CheckedContinuation` instances across `Entry` records — each
   Entry must own exactly one continuation.
4. Adding a "single-awaiter-per-fd" precondition anywhere in the completions
   path.

### Revalidation

Before any Option C work modifies either actor, re-run the grep in
"Current evidence" against the then-current commit. If it still returns no
matches and the four referenced sites still have the stated shape, the
invariant holds. If not, the architectural-symmetry argument for Option C
must be re-examined.

A regression guard suitable for CI is intentionally absent: the invariant is
structural, not behavioural, and a lint-style check (`grep Async.Channel
Sources/IO\ Completions/` rejecting matches) would be fragile under
unrelated import reorganizations. This section, the grep, and the four file
citations are the primary defense.
